# 판단 기준

## 역할 분리

다음 중 하나라도 해당하면 Planner, Architect, Executor와 Reviewer를 분리한다.

- 사용자가 역할 분리, 위임, 하위 에이전트, 병렬 작업 또는 squad를 요청했다.
- 티켓, 계약, acceptance criteria 또는 명시적인 승인 게이트가 있다.
- 데이터 모델, 마이그레이션, 인증, 권한, 결제, 인프라, 배포 또는 운영 동작을 변경한다.
- 여러 앱, 패키지, 서비스, 저장소 또는 담당 영역에 걸친다.
- 핵심 사용자 흐름이나 외부에 보이는 동작을 변경한다.
- 스크린샷, 증거, 리뷰 보고서 또는 감사 가능한 완료 기록이 필요하다.
- 잘못된 구현의 비용이 높아 독립적인 설계 또는 리뷰가 유용하다.

## 한 세션에서 처리

다음처럼 작고 로컬 범위가 명확하면 역할 분리를 생략한다.

- 질문에 답하거나 코드를 읽고 요약한다.
- 오탈자나 작은 문서 문제를 고친다.
- 동작이 명확한 단일 파일을 수정한다.
- 실패를 쉽게 발견할 수 있고 영향이 작다.

## Architect를 생략할 수 있는 조건

다음 조건을 모두 만족할 때 Architect 단계를 생략할 수 있다.

- 주변 코드에서 구현 패턴이 이미 명확하다.
- 수정 범위가 작고 하나의 단위로 묶인다.
- 실패를 쉽게 발견할 수 있고 영향이 작다.
- Planner가 관련 파일과 저장소 규칙을 확인했다.

## Reviewer를 생략하지 말아야 하는 경우

다음 중 하나라도 해당하면 Reviewer를 사용한다.

- PR을 열거나 병합한다.
- 티켓 또는 계약을 완료로 표시한다.
- UI 동작이 바뀐다.
- 데이터, 인증, 결제, 인프라, 배포 또는 운영 동작이 바뀐다.
- 여러 Executor가 변경에 참여했다.
- 사용자가 중요, 긴급, 위험 또는 리뷰를 명시했다.

## 멈추고 확인할 경우

- 필요한 자격 증명, 접근 권한 또는 파일이 없다.
- 계약이 필요한데 없거나, 요청이 계약·저장소 규칙과 충돌한다.
- 금지된 범위나 파일을 수정해야 한다.
- 기존 로컬 변경과 새 변경이 겹치고 소유권이 불명확하다.
- 다음 단계가 운영 데이터, 결제, 배포 또는 외부 사용자에게 영향을 준다.
- 검증에 장기 실행 서비스, 유료 리소스 또는 사용자가 승인하지 않은 파괴적 작업이 필요하다.
- 고위험 작업의 에이전트 정책이 없거나 비용 상한을 확인할 수 없다.

## 모델과 에이전트 선택

- 역할에는 정책 파일에서 읽은 고정 `model_id`와 `capability`, `effort_ceiling`, `cost_ceiling`을 전달한다.
- 프로젝트 정책이 없으면 최초 실행 시 AI 도구, 고정 모델 ID, 기본 effort와 비용 상한을 한 번 묻고 `.agents/agent-policy.md`에 저장한다.
- `quality`는 프로젝트에서 허용한 품질 프로필이며 최신 또는 최고가 모델을 자동 선택한다는 뜻이 아니다.
- `xHigh`는 Architect처럼 정책에 명시된 역할에서만 사용한다.
- `critical` 또는 `approved` 비용 상한은 사용자나 프로젝트 정책의 명시적 승인이 있을 때만 사용한다.
- 새 모델·effort 등급이 추가되어도 정책에 없으면 자동으로 선택하지 않는다.
- 실제 도구가 어떤 모델과 effort를 적용했는지 확인할 수 없으면 선택 결과를 추측하지 말고 확인 불가로 보고한다.
- 역할 수를 늘리기보다 작업 결합도가 높으면 적은 수의 강한 역할로 처리한다.
- Planner가 최종 통합, 검증과 사용자-facing 결론을 책임진다.

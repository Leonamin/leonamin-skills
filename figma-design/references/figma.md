# Figma MCP 워크플로

Figma를 읽거나 쓰는 작업에서만 이 참고 자료를 사용한다.

## 사전 조건

- 쓰기 도구를 호출하기 전에 환경의 Figma MCP 스킬이 요구하는 사전 조건을 따른다.
- 새 원시 컴포넌트를 만들기 전에 기존 Figma 라이브러리와 컴포넌트를 검색·확인한다.
- 저장소의 design-product/ 파일을 디자인 브리프와 가드레일의 원천으로 사용한다.
- Figma를 이미지 캔버스로 사용하지 않는다. 편집 가능한 네이티브 구조를 만든다.

## 필수 흐름

1. design-product 컨텍스트를 읽는다.
2. 화면 또는 컴포넌트 유형을 분류한다.
3. IA·컴포넌트 계획을 만든다.
4. 사용할 Figma 페이지를 확인한다.
   - 00 Brand Board
   - 01 Foundations
   - 02 Components
   - 03 Product Patterns
   - 04 Screens
   - 99 Rejected Patterns
5. 가능하면 토큰을 Figma 변수에 연결한다.
6. 이름 있는 프레임과 오토 레이아웃으로 생성·갱신한다.
7. 제품 맥락이 드러나는 컴포넌트 이름을 사용한다.
8. 결과를 감사한다.
9. 디자인 memory를 갱신한다.

## 네이티브 Figma 요구사항

- 프레임, 오토 레이아웃, 제약 조건, 변수, 스타일, 컴포넌트와 변형을 사용한다.
- 레이어는 시각적 우연이 아니라 도메인 역할에 따라 이름 붙인다.
- 일시적인 참고가 아니라면 화면을 납작한 스크린샷으로 만들지 않는다.
- 일회성 색상과 radius 값을 피한다.
- 유용하다면 거부한 탐색안은 99 Rejected Patterns에 보관한다.

## 레이아웃 정책

오토 레이아웃은 모든 시각 노드에 기계적으로 적용하는 규칙이 아니라, 구조적 관계를 편집 가능한 제약으로 표현하는 기본 수단이다. `x`와 `y` 속성은 오토 레이아웃이 계산한 결과에도 존재하므로 속성의 존재 자체를 실패로 판단하지 않는다. 대신 일반 흐름의 자식이 `layoutPositioning = ABSOLUTE`인지, 부모가 구조적 관계를 오토 레이아웃으로 소유하는지를 확인한다.

### 기본 규칙

- Root Frame은 화면의 정상적인 콘텐츠 흐름을 소유하므로 오토 레이아웃을 적용한다. Figma 페이지 위에 Root Frame을 놓기 위한 캔버스 좌표는 허용한다.
- 순서, 정렬, 간격 또는 리사이징 관계를 소유하는 작성 프레임은 오토 레이아웃을 사용한다.
- 일반 콘텐츠 흐름의 자식은 `layoutPositioning = ABSOLUTE`로 두지 않는다.
- 가로를 채워야 하는 입력창, 주요 버튼, 섹션은 부모에 추가한 다음 가로 크기를 `Fill container`로 설정한다.
- 내부 콘텐츠 크기를 따라야 하는 버튼, 칩, 레이블은 필요한 축을 `Hug contents`로 설정한다.
- 줄바꿈 텍스트는 명시적인 가로 동작과 콘텐츠에 따른 세로 동작을 함께 설정하고, 폭이 0에 가깝게 축소되지 않았는지 확인한다.
- 형제 사이 간격은 `item spacing`, 컨테이너 안쪽 여백은 `padding`으로 표현한다.
- 디자인 시스템에 대응하는 spacing 변수가 있으면 gap과 padding에 연결한다.
- 연결된 라이브러리 인스턴스의 내부 구조는 정책 검사의 소유 범위에서 제외한다. 인스턴스를 detach하거나 내부 프레임을 다시 만들지 않는다.

### 예외가 필요한 경우

다음처럼 공간 위치 자체가 디자인 의미인 경우에는 절대 배치 또는 자유 배치 프레임을 허용한다.

- Staggered Grid 또는 Masonry처럼 행 정렬이 의도적으로 불규칙한 배치
- 배지, 툴팁, 플로팅 액션, 팝오버처럼 다른 요소와 겹치는 오버레이
- 차트, 지도, 노드 캔버스, 타임라인처럼 좌표가 데이터를 표현하는 화면
- 마스크, 클리핑, 장식 그래픽처럼 시각 합성이 목적인 레이어

예외는 다음 조건을 모두 만족해야 한다.

1. 일반 오토 레이아웃, wrap 또는 grid로 같은 의미와 리사이징 동작을 보존할 수 없는지 먼저 확인한다.
2. 화면 전체가 아니라 자유 배치가 필요한 가장 작은 하위 프레임에 격리한다.
3. 프레임 이름을 `Layout Exception/<reason>` 형식으로 지정한다.
4. 예외 프레임 자체는 상위 오토 레이아웃의 정상 흐름에 참여시킨다.
5. 절대 배치 자식은 예외 프레임 안에서만 사용하고, 프레임의 고정 크기·최소 크기·클리핑·브레이크포인트 동작을 명시한다.
6. 작업 결과에 예외 이유, 영향 범위, 검증한 크기를 기록한다.

### Staggered Grid 선택 기준

1. 네이티브 grid 또는 wrap이 카드 순서와 리사이징을 보존하면 이를 우선한다.
2. 열 단위 읽기 순서가 허용되면 `Horizontal Auto Layout > Vertical Column Auto Layout > Card` 구조로 만든다. 카드와 열은 `Fill container` 또는 합의된 고정 폭을 사용한다.
3. 정확한 시각 순서, 겹침 또는 좌표가 제품 의미라면 `Layout Exception/Staggered Grid` 프레임 안에서만 자유 배치한다. 절대 배치 자식은 자동 크기 계산에서 제외되므로 예외 프레임의 높이와 브레이크포인트별 동작을 명시한다.
4. 콘텐츠 추가·삭제·번역·긴 텍스트 상황에서 카드가 겹치거나 잘리지 않는지 확인한다.

### 검증 기준

- 작성한 레이아웃 컨테이너에서 `layoutMode = NONE`인 노드를 찾는다. 예외로 분류되지 않았다면 실패다.
- Root Frame 내부의 `layoutPositioning = ABSOLUTE` 노드를 찾는다. `Layout Exception/` 하위가 아니면 실패다.
- `Fill container`와 `Hug contents`가 의도한 노드에 적용됐는지 이름과 역할별로 확인한다.
- gap과 padding이 지정 값 또는 변수에 연결됐는지 확인한다.
- 예외 프레임은 최소 한 개 이상의 목표 크기와 콘텐츠 변화 조건에서 스크린샷으로 검증한다.
- 전체 화면뿐 아니라 주요 섹션도 따로 렌더링해 텍스트 잘림, 겹침, 잘못된 순서와 빈 공간을 확인한다.
- 검사 결과에는 정상 오토 레이아웃 컨테이너, 승인된 예외, 미분류 절대 배치를 구분해 기록한다.

## Figma 도구가 없을 때

파일이 갱신되었다고 가장하지 않는다. 대신 다음을 제공한다.

- IA·컴포넌트 계획
- Figma 페이지·프레임·컴포넌트 구조
- 토큰과 변수 매핑
- 감사 체크리스트
- Figma MCP가 가능할 때 실행할 정확한 다음 명령 또는 도구 작업
